Visão Geral

VulnGate é uma ferramenta CLI que utiliza o Trivy como motor principal para escanear vulnerabilidades, segredos e misconfigurations em projetos.

Stacs suportados:

  • Java — Maven, Gradle, JAR, WAR, EAR, Java 8, 11, 21, 25
  • Node.js — npm, yarn
  • React / ReactJS
  • Vue.js
  • Filesystem — diretórios e repositórios locais
  • Git repositories — clona e escaneia automaticamente

Pré-requisitos

⚙

Go 1.22+

Necessário apenas para compilar a partir do código-fonte.

🔍

Trivy

O Trivy deve estar instalado e disponível no PATH. O VulnGate verifica automaticamente a presença do Trivy antes de cada scan.

Instalação

Instalando o Trivy

# Linux (Debian/Ubuntu)
$ sudo apt-get update
$ sudo apt-get install -y trivy

# Linux (RHEL/CentOS)
$ sudo yum install trivy

# macOS
$ brew install trivy

# Windows (Chocolatey)
$ choco install trivy

# Go install
$ go install github.com/aquasecurity/trivy/cmd/trivy@latest

Compilando o VulnGate

# Clonar o repositório
$ git clone https://github.com/leonardo-matheus/vulnscan.git
$ cd vulnscan

# Compilar
$ go build -o vulngate .

# Ou instalar localmente
$ go install .
Verificação: Execute vulngate version para confirmar que tanto o VulnGate quanto o Trivy estão instalados corretamente.

Comandos

📂

vulngate scan fs [path]

Escaneia um diretório do filesystem local.

$ vulngate scan fs .
$ vulngate scan fs /caminho/para/projeto
$ vulngate scan fs ./meu-app-java --severity HIGH,CRITICAL
$ vulngate scan fs ./meu-app-node --include-dev-deps
$ vulngate scan fs . --format sarif --output results.sarif
📦

vulngate scan repo [git-url]

Clona um repositório Git temporariamente e escaneia-o. O repositório é removido automaticamente ao final.

$ vulngate scan repo https://github.com/org/repo
$ vulngate scan repo git@github.com:org/repo.git
$ vulngate scan repo https://github.com/org/repo --severity HIGH,CRITICAL
$ vulngate scan repo https://github.com/org/repo --format sarif --output report.sarif
ℹ️

vulngate version

Exibe a versão do VulnGate e do Trivy instalado.

$ vulngate version

Flags Globais

Flag Padrão Descrição
--severity HIGH,CRITICAL Níveis de severidade para reportar (LOW, MEDIUM, HIGH, CRITICAL)
--format table Formato de saída (table, json, sarif)
--output — Caminho do arquivo de saída
--exit-code 1 Exit code quando vulnerabilidades são encontradas
--ignore-unfixed false Ignorar vulnerabilidades sem correção disponível
--include-dev-deps false Incluir dependências de desenvolvimento (Node.js/React/Vue)
--scanners vuln,secret,misconfig Tipos de scanner (vuln, secret, misconfig)
--timeout 10m Timeout do scan
--debug false Habilitar saída de debug (mostra comando Trivy executado)

Exemplos com Flags

# Scan com todas as severidades
$ vulngate scan fs . --severity LOW,MEDIUM,HIGH,CRITICAL

# Scan apenas de vulnerabilidades (sem secrets e misconfig)
$ vulngate scan fs . --scanners vuln

# Export para JSON
$ vulngate scan fs . --format json --output report.json

# Export para SARIF (integração com GitHub Security)
$ vulngate scan fs . --format sarif --output trivy-results.sarif

# Ignorar vulnerabilidades sem fix
$ vulngate scan fs . --ignore-unfixed

# Modo debug
$ vulngate scan fs . --debug

# Timeout personalizado
$ vulngate scan fs . --timeout 30m

Exemplos por Tecnologia

☕ Java (Maven)

# Scan projeto Maven
$ vulngate scan fs ./my-java-app

# Apenas vulnerabilidades
$ vulngate scan fs ./my-java-app \
  --scanners vuln

🐘 Java (Gradle)

# Scan projeto Gradle
$ vulngate scan fs ./my-gradle-app

# Scan JAR específico
$ vulngate scan fs ./target/app.jar

📦 Java (WAR/EAR)

# Scan WAR
$ vulngate scan fs ./target/app.war

# Scan EAR
$ vulngate scan fs ./target/app.ear

🟢 Node.js

# Com dev dependencies
$ vulngate scan fs ./my-node-app \
  --include-dev-deps

# Sem dev dependencies
$ vulngate scan fs ./my-node-app

⚛️ React

# Scan React com dev deps
$ vulngate scan fs ./my-react-app \
  --include-dev-deps

# SARIF para GitHub
$ vulngate scan fs ./my-react-app \
  --include-dev-deps \
  --format sarif \
  --output results.sarif

💚 Vue.js

# Scan Vue.js com dev deps
$ vulngate scan fs ./my-vue-app \
  --include-dev-deps

# Com severidade específica
$ vulngate scan fs ./my-vue-app \
  --include-dev-deps \
  --severity HIGH,CRITICAL
Dica: A flag --include-dev-deps é especialmente importante para projetos Node.js, React e Vue.js, pois muitas vulnerabilidades estão em dependências de desenvolvimento (ex: Webpack, Babel, ESLint).

Uso em GitHub Actions

Workflow Completo

name: Security Scan

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up Go
        uses: actions/setup-go@v5
        with:
          go-version: '1.23'

      - name: Install Trivy
        run: |
          sudo apt-get update
          sudo apt-get install -y trivy

      - name: Build VulnGate
        run: go build -o vulngate .

      - name: Run scan
        run: |
          ./vulngate scan fs . --format sarif --output trivy-results.sarif

      - name: Upload SARIF
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: trivy-results.sarif

SARIF Upload para GitHub Security

      - name: Run Trivy
        run: |
          ./vulngate scan fs . --format sarif --output results.sarif --severity HIGH,CRITICAL

      - name: Upload to GitHub Security
        uses: github/codeql-action/upload-sarif@v3
        if: always()
        with:
          sarif_file: results.sarif
Importante: Use if: always() no step de upload para garantir que o SARIF seja enviado mesmo quando o scan encontra vulnerabilidades (exit code != 0).

Exit Codes

Código Descrição
0 Nenhuma vulnerabilidade encontrada
1 Vulnerabilidades encontradas (padrão)
2 Erro de execução do Trivy

Controle via Flag

# Falhar no pipeline se encontrar HIGH/CRITICAL
$ vulngate scan fs . --exit-code 1

# Sempre sucesso (apenas reportar)
$ vulngate scan fs . --exit-code 0

# Exit code customizado
$ vulngate scan fs . --exit-code 2
CI/CD: O exit code é compatível com pipelines de CI/CD. Se --exit-code 1 e vulnerabilidades forem encontradas, o pipeline falha automaticamente.
